# Single-host tenant access

This project is deployed once and shared by every company. There is one admin application, one storefront application, and one API. Tenant isolation is enforced by the backend; changing a browser path cannot grant access to another company.

## Admin access

Each company receives an admin entry URL:

```text
https://admin.example.com/workspace/{workspace-code}/login
```

The workspace code only preselects the company on the login screen. It is **not** authorization. After a successful login, the API issues a signed JWT containing the authenticated user's tenant claim. Every authenticated admin request is resolved from that signed tenant claim, and browser headers or URLs cannot replace it.

A user can therefore work only in the company to which that account belongs. Platform administrators use the same shared admin application and receive the separate platform-administrator claim.

## Public storefront access

A published public store receives a random opaque URL:

```text
https://shop.example.com/store/{storefront-key}
```

The storefront key is generated by the server, is not based on the company name or numeric ID, and can be rotated from Company Settings or Platform → Tenant Companies. Rotating it immediately invalidates the old public URL.

The SPA reads the key from the path and sends it to the API as:

```http
X-Storefront-Key: {storefront-key}
```

The API resolves only an active, subscribed, published tenant whose stored key matches. Customer JWT sessions are also tenant-bound, so an authenticated customer cannot switch companies by editing the store path.

A public storefront URL is a routing identifier, not a password. Anyone who receives a public store link may view that published store, just as with a normal public e-commerce URL.

## Private storefront and previews

A tenant can set its storefront to **Private** or unpublished. Private/unpublished stores do not have a public URL. An authorized tenant or platform administrator can generate a short-lived preview URL:

```text
https://shop.example.com/preview/{signed-token}
```

The preview token is signed by the API, contains the tenant ID and current storefront key, and expires. Rotating the storefront key invalidates existing preview links as well.

The SPA forwards the token as:

```http
X-Storefront-Preview: {signed-token}
```

## Shared-host deployment checklist

1. Deploy one admin SPA and one storefront SPA.
2. Configure **Platform Settings → Admin base URL** and **Storefront base URL**.
3. Point `VITE_API_BASE_URL` in both SPAs to the same API deployment.
4. Configure the API CORS allow-list with only the shared admin/storefront origins.
5. Configure SPA fallback routes for `/workspace/*`, `/store/*`, and `/preview/*` in IIS or the reverse proxy.
6. Use **Tenant Companies** or **Company Settings** to publish a store, copy its URL, generate a private preview, or rotate its key.

No tenant query string, tenant subdomain, custom domain, or second deployment is required.

## Notification lifecycle

Tenant administrators may delete one notification or clear all notifications. Manual deletion hides the item immediately by soft deletion. Every six hours, the cleanup worker permanently deletes admin notifications older than the tenant's configured retention period. The retention setting is available under Company Settings and supports 1–365 days.
